This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

GitOps repository for the **jellybuntu** Kubernetes cluster, managed by **Flux CD v2.8.1**. All cluster state is declared in this repo — Flux automatically reconciles changes pushed to the `main` branch.

Repository: `git@github.com:SilverDFlame/jellybuntu-helm`

## Validation Commands

There is no CI pipeline. Validate manifests locally before pushing:

```bash
# Validate all kustomize overlays build cleanly
kubectl kustomize clusters/jellybuntu/
kubectl kustomize clusters/jellybuntu/infrastructure/
kubectl kustomize clusters/jellybuntu/infrastructure/sources/
kubectl kustomize clusters/jellybuntu/infrastructure/controllers/

# Validate individual YAML files
kubectl apply --dry-run=client -f <file>

# Check Flux reconciliation status (requires cluster access)
flux get all -A
flux get kustomizations
flux get helmreleases -A
```

## Architecture

### Layered GitOps Structure

```
clusters/jellybuntu/
├── flux-system/              # Flux bootstrap (generated, rarely edited manually)
│   ├── gotk-components.yaml  # Flux controllers (~6400 lines, regenerated by flux bootstrap)
│   └── gotk-sync.yaml        # GitRepository + root sync Kustomization
├── infrastructure.yaml       # Flux Kustomization → infrastructure/
├── infrastructure/
│   ├── sources/              # HelmRepository definitions
│   └── controllers/          # HelmRelease + Namespace definitions
├── net.yaml → net/           # MetalLB IP pool config
├── media.yaml → media/       # Media apps (placeholder)
├── gpu.yaml → gpu/           # GPU workloads (placeholder)
└── ops.yaml → ops/           # Observability (placeholder)
```

Each `{layer}.yaml` at the cluster root is a **Flux Kustomization** resource that points to the corresponding `{layer}/` directory. The directory contains a standard `kustomization.yaml` listing its resources.

### Dependency Chain

```
infrastructure (foundation, no dependencies)
  ↓ dependsOn
media, gpu, net, ops (all depend on infrastructure)
```

## Key Conventions

### Namespace Management

Namespaces **must be pre-created** before HelmReleases reference them. Infrastructure namespaces live in `infrastructure/controllers/namespaces.yaml`. Application layer namespaces (e.g., `media`, `matrix`) are defined in their own layer directory as `namespace.yaml` and listed first in the layer's `kustomization.yaml`.

### Naming Patterns

- Infrastructure/controller namespaces: `{service}-system` (e.g., `metallb-system`, `traefik-system`, `nfs-system`)
- Application workload namespaces: bare names (e.g., `media`, `matrix`)
- All resources labeled: `app.kubernetes.io/part-of: jellybuntu`
- Flux Kustomizations named by layer: `infrastructure`, `media`, `gpu`, `net`, `ops`

### API Versions

- Flux Kustomizations: `kustomize.toolkit.fluxcd.io/v1`
- HelmReleases: `helm.toolkit.fluxcd.io/v2`
- HelmRepositories: `source.toolkit.fluxcd.io/v1`

### Adding a New HelmRelease

1. Add HelmRepository source in `infrastructure/sources/` and register in its `kustomization.yaml`
2. Create Namespace in `infrastructure/controllers/namespaces.yaml`
3. Add HelmRelease in `infrastructure/controllers/` (or appropriate layer) and register in its `kustomization.yaml`
4. All Kustomizations use `prune: true` — removed resources are garbage-collected

### Sync Configuration

- GitRepository watches `main` branch
- All Flux Kustomizations use **1-hour** reconciliation intervals
- `gotk-sync.yaml` uses **10-minute** interval for the root sync

## Files to Avoid Editing Manually

- `flux-system/gotk-components.yaml` — regenerated by `flux bootstrap`; manual edits will be overwritten
